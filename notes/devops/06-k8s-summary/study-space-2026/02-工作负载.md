# 02 · 工作负载 Workload

工作负载 = **管理 Pod 的控制器**。选哪个取决于应用形态:

```
                    ┌─ 无状态(Web/API) ──────→ Deployment
   要长期运行? ──是─┼─ 有状态(DB/有身份) ────→ StatefulSet
                    └─ 每节点一份(日志/监控)→ DaemonSet
   跑完就结束? ──是─┬─ 一次性任务 ───────────→ Job
                    └─ 定时任务 ─────────────→ CronJob
```

## 1. Pod
- 最小调度单位,含 1+ 容器,共享网络与卷。
- 裸 Pod **无自愈**,所以用控制器间接管理。
- 多容器 Pod 何时合理?三连问:必须同机?是一个整体?需一起扩缩容?——典型是 **Sidecar 模式**(主容器 + 日志/代理边车)。

## 2. ReplicaSet(RS)
- 唯一职责:**始终保持 N 个相同 Pod 运行**(调和循环作用在 Pod 数量上)。
- 靠**标签选择器**认领 Pod,不是按名字。
- ⚠️ 坑:改了 selector 或 Pod 标签使其不再匹配 → Pod 脱管成孤儿,RS 又补建一批 → 多出 Pod。
- 比老的 ReplicationController(RC,已废弃)强在支持 `matchExpressions`(In/NotIn/Exists)。
- **你几乎从不直接用 RS,而是用 Deployment 来管它。**

## 3. Deployment —— 无状态标配
层级:`Deployment → ReplicaSet → Pod`。

- 改 Pod 模板(`.spec.template`)→ 新建一个 RS → 流量从旧 RS 逐步挪到新 RS(滚动更新)。旧 RS 保留(默认 10 个)用于回滚 → 这就是"为什么出现两个 RS"。
- **滚动更新策略**(零停机):
```yaml
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxUnavailable: 25%   # 更新时最多 25% 不可用
    maxSurge: 25%         # 最多额外多起 25%
```
- `apps/v1` 里 selector **创建后不可改**,要改只能删重建。
- **Pause/Resume**:想改多项配置只触发一次发布,或发一部分先观察 → 先 pause,改完 resume。
- 常用命令:
```bash
kubectl rollout status deployment my-app
kubectl rollout history deployment my-app
kubectl rollout undo deployment my-app --to-revision=1
kubectl scale --replicas=3 deployment my-app
kubectl autoscale deployment my-app --min=10 --max=15 --cpu-percent=80   # HPA
```

## 4. StatefulSet —— 有状态应用
用于数据库、Kafka、etcd 等。与 Deployment 的本质区别是"身份"。三大保证:

1. **稳定有序的网络标识**:Pod 名固定有序(`mysql-0/1/2`),每个有固定 DNS 域名 `mysql-0.mysql-svc.ns.svc.cluster.local`,重新调度 IP 变了域名不变。
2. **稳定的持久化存储**:`volumeClaimTemplates` 给每个 Pod 单独建 PVC,`mysql-0` 永远绑它那块盘。⚠️ `hostPath` 在跨节点调度后丢数据,必须用集群存储(PV/PVC + Ceph/NFS)。
3. **有序部署/伸缩**:严格 `0→1→2` 创建,逆序删除(适合主从,先起主)。

**为什么必须配 Headless Service**:有状态应用要**点名访问某个具体 Pod**。Headless Service(`clusterIP: None`)不分配 VIP,DNS 直接返回所有 Pod IP,并给每个 Pod 单独 DNS 记录。

| | ClusterIP Service | Headless Service |
|---|---|---|
| clusterIP | 有 VIP | None |
| DNS 返回 | 一个 VIP | 所有 Pod 的 IP 列表 |
| 用途 | 无状态负载均衡 | 有状态点名访问 |

## 5. DaemonSet —— 每节点一份
- 保证每个(符合条件的)节点恰好跑一个 Pod;新节点自动部署。
- 用途:日志(Fluentd/Vector)、监控(Node Exporter)、网络 CNI(Calico/Cilium)、存储守护进程。
- 不设副本数,副本数 = 节点数。

## 6. Job —— 一次性任务
```yaml
spec:
  completions: 5     # 要成功 5 次
  parallelism: 2     # 同时最多 2 个
  backoffLimit: 4    # 失败重试上限
  ttlSecondsAfterFinished: 3600   # 完成后自动清理
```

## 7. CronJob —— 定时任务(K8s 版 crontab)
```yaml
spec:
  schedule: "0 2 * * *"          # 每天 2:00
  concurrencyPolicy: Forbid       # 上次没跑完不再起
  successfulJobsHistoryLimit: 3   # 控制历史数,防 etcd 膨胀
  jobTemplate:
    spec:
      template: ...               # 内嵌一个 Job
```
