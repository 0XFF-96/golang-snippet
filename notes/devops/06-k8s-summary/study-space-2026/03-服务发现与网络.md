# 03 · 服务发现与网络

> 核心问题:Pod 会死会重建、IP 一直变,**怎么找到并稳定访问它们?**

## 三层 IP(排错先分清断在哪层)
- **Pod IP**:每个 Pod 一个,集群内可路由,Pod 重建即变。
- **Cluster IP(Service IP)**:稳定虚拟 IP,kube-proxy 转发到后端 Pod。
- **Node IP**:节点真实 IP。

## 1. Service —— 集群内稳定入口
- 用**标签选择器**把一组 Pod 框成逻辑组,给固定 IP + 端口,负载均衡转发。Pod 来去,Service IP 不变。

### Endpoints —— 幕后角色
- Service 不直接连 Pod,中间隔 **Endpoints**,记录当前"健康且匹配"的 Pod IP 列表。
- 🔴 **排错第一招**:`kubectl get endpoints <svc>`。**Endpoints 为空 = selector 没匹配上 Pod,或 Pod readiness 未就绪**。

### kube-proxy 三种模式
| 模式 | 原理 | 说明 |
|------|------|------|
| userspace | 用户态转发 | 老、慢、废弃 |
| **iptables** | 内核 iptables NAT | 默认,随机负载均衡 |
| **IPVS** | 内核 IPVS | 大规模性能好,支持多种 LB 算法 |

### Service 四种类型
```yaml
spec:
  type: ClusterIP      # ① 默认:仅集群内访问
  # type: NodePort     # ② 每节点开端口(30000-32767),外部 NodeIP:NodePort 访问
  # type: LoadBalancer # ③ 云厂商外部 LB,生产暴露常用
  # type: ExternalName # ④ 不选 Pod,只给 DNS CNAME(如指向外部 RDS)
```
> 内部 Pod 访问外部 MySQL RDS → 用 `ExternalName` 或手动建 Endpoints,把外部地址包装成集群内 Service 名。

### 三个 Port 的区别(高频面试题)
```
外部用户 → [NodePort:30080] → [Service port:80] → [targetPort:8080 = 容器监听端口]
```
- **targetPort**:Pod 里容器实际监听的端口(你程序 listen 的)。
- **port**:Service 在集群内暴露的端口,集群内用 `ClusterIP:port`。
- **nodePort**:节点上开的端口,集群外用 `NodeIP:nodePort`(仅 NodePort/LoadBalancer 有)。

## 2. 服务发现的两种方式
- **① 环境变量**:Pod 启动时注入已存在 Service 的信息。🔴 局限:**有顺序依赖**,Service 必须先于 Pod 创建,后建的注入不进去 → 基本不用。
- **② DNS(CoreDNS)—— 标准方式**:每个 Service 自动获 DNS 名:
```
myapp-svc                              # 同 namespace 用短名
myapp-svc.default                      # 跨 namespace
myapp-svc.default.svc.cluster.local    # 完整 FQDN
```
短名能用是因为 `/etc/resolv.conf` 的 search 域。

## 3. Ingress —— 集群外 HTTP 入口
- 痛点:每个服务都用 LoadBalancer → 要买 N 个云 LB,贵且乱。
- Ingress = **集群入口的七层路由器**,一个入口按域名/路径分发到不同 Service。本质把"改 nginx 配置"声明化、自动化。

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
spec:
  rules:
    - host: app.example.com
      http:
        paths:
          - path: /api
            pathType: Prefix
            backend: { service: { name: api-svc, port: { number: 80 } } }
          - path: /
            pathType: Prefix
            backend: { service: { name: web-svc, port: { number: 80 } } }
  tls:
    - hosts: [app.example.com]
      secretName: app-tls          # tls 类型 Secret
```
- ⚠️ Ingress 只是规则,真正干活的是 **Ingress Controller**(nginx-ingress / Traefik / 云 ALB),必须先装一个。
- 分工:**Service 管 L4 集群内访问;Ingress 管 L7 集群外路由**。Ingress 后面接的还是 Service。

## 完整数据流
```
外部用户
  → DNS 解析 app.example.com 到 Ingress 的 LB IP
  → Ingress Controller(按域名/路径路由)
  → Service(ClusterIP,负载均衡)
  → Endpoints(健康 Pod 列表)
  → Pod(你的程序)
```
