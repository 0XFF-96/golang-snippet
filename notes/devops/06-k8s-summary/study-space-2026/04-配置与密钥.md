# 04 · 配置与密钥(ConfigMap & Secret)

## 为什么需要 ConfigMap —— 解耦
硬编码配置的问题:不能复用(三套环境要打三个镜像)、改配置要重新 build、配置和代码绑死。
**ConfigMap 核心价值:把配置从镜像中解耦**。同一镜像挂不同 ConfigMap → 跑在不同环境。

## 创建 ConfigMap 的四种方式
```bash
# ① 字面量
kubectl create configmap special-config \
  --from-literal=special.level=very --from-literal=special.type=charm
# ② 单文件(key=文件名,value=文件内容)
kubectl create configmap my-config --from-file=app.properties
# ③ 整个目录(每个文件一个 key)
kubectl create configmap my-config --from-file=./config-dir/
# ④ env 文件(每行 key=value 一个 key)
kubectl create configmap my-config --from-env-file=app.env
```

## 在 Pod 里消费的三种姿势
**A. 单个环境变量**
```yaml
env:
  - name: SPECIAL_LEVEL
    valueFrom:
      configMapKeyRef:
        name: special-config
        key: special.level
        optional: true   # 不加这个,ConfigMap 不存在时容器启动失败
```
**B. 一次性全部变环境变量**
```yaml
envFrom:
  - configMapRef: { name: special-config }
```
**C. 挂载成文件(最常用)**
```yaml
volumes:
  - name: config-vol
    configMap:
      name: special-config
      items:
        - { key: nginx.conf, path: nginx.conf }   # 用 path 指定文件名
containers:
  - volumeMounts:
      - { name: config-vol, mountPath: /etc/nginx/conf.d }
```

## 热更新 —— 有坑
- **卷挂载**:改 ConfigMap,挂进去的文件会自动更新(默认延迟约 1 分钟)。但**应用自己得监听文件变化(inotify)**,否则进程没重读 = 没更新。
- **环境变量方式不热更新**:env 启动时固定,改了必须重启 Pod。
- 生产技巧:在 Pod 模板加 `checksum/config` 注解(配置内容 hash),配置一变 hash 变 → Pod 滚动重启。Helm/Argo 都这么干。

## Secret —— 用于敏感数据
| 类型 | 用途 | 命令 |
|------|------|------|
| **generic (Opaque)** | 密码/token/密钥 | `kubectl create secret generic db --from-literal=password=123` |
| **docker-registry** | 拉私有镜像凭证 | `kubectl create secret docker-registry regcred --docker-server=... --docker-username=...` |
| **tls** | TLS 证书(给 Ingress) | `kubectl create secret tls my-tls --cert=tls.crt --key=tls.key` |

**关键认知:**
- 🔴 **Secret 默认只是 base64,不是加密!** 任何能读的人 `base64 -d` 就解开。真加密要开 **etcd encryption at rest** + RBAC 限制读取。
- Secret 大小上限 **1 MB**。
- 生产更安全:**External Secrets Operator / Sealed Secrets / Vault**,绝不把明文 Secret 提交进 Git。

## default-token 与 ServiceAccount
- 每个 Pod 默认挂载所属 SA 的凭证到 `/var/run/secrets/kubernetes.io/serviceaccount/`,含 `ca.crt`、`namespace`、`token`。
- 作用:Pod 内程序凭 token 访问 API Server(如 client-go 写的 operator)。
- 不需访问 API 的 Pod 应关掉:`automountServiceAccountToken: false`(减少攻击面)。
- 让某 SA 下所有 Pod 自动带拉镜像凭证:把 `imagePullSecrets` 挂到 SA 上:
```bash
kubectl patch serviceaccount default \
  -p '{"imagePullSecrets":[{"name":"regcred"}]}'
```
