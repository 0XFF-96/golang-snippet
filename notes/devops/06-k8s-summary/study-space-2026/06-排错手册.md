# 06 · 排错手册

## 通用调试命令
```bash
kubectl get events --sort-by=.lastTimestamp -A      # 集群里到底发生了什么
kubectl describe pod <pod>                            # 事件在最下方
kubectl logs <pod> --previous                         # 崩溃容器的上一次日志(排 CrashLoop 关键)
kubectl top pods / nodes                              # 实时资源用量
kubectl get pod <pod> -o yaml | grep -A5 lastState    # OOMKilled? 退出码?
kubectl debug -it <pod> --image=busybox --target=<container>   # 临时调试容器
kubectl port-forward pod-xxx 8080:80                  # 本地端口转发,本地调试神器
```

## Pod 常见状态含义
| 状态 | 含义 | 处理 |
|------|------|------|
| `CrashLoopBackOff` | 启动后就退出 | 看 `logs --previous` |
| `ImagePullBackOff` | 镜像名/标签错或拉取认证失败 | 查镜像名、imagePullSecrets |
| `Pending` | 无法调度 | `describe` 看:资源不足 / 污点 / PVC 未绑定 |
| `OOMKilled`(退出码 137) | 内存超限 | 调大 limit 或修内存泄漏 |
| `Evicted` | 节点压力(磁盘/内存) | 看节点资源 |

## 1. 网络排查(由内到外)
```bash
kubectl get pod -o wide            # ① Pod 起来了吗?看节点、Pod IP
kubectl get endpoints my-svc       # ② Service 找到后端了吗?空 = selector 没匹配或未就绪
kubectl run tmp --rm -it --image=nicolaka/netshoot -- bash   # ③ 临时 Pod 测连通
  curl http://my-svc:80
  nslookup my-svc
```
> 推荐 `nicolaka/netshoot` 镜像,自带 dig/curl/tcpdump/netstat 全套,比 `apt-get install` 方便。

## 2. DNS 排查
Service DNS 格式:`<service>.<namespace>.svc.cluster.local`
```bash
kubectl get pods -n kube-system -l k8s-app=kube-dns   # CoreDNS 活着没
kubectl exec -it mypod -- nslookup kubernetes.default # 进 Pod 测解析
kubectl exec -it mypod -- cat /etc/resolv.conf        # 看 DNS 配置
```
常见坑:
- **`ndots:5`**:访问外部域名会先拼 5 次集群内后缀全失败才查真正的 → 拖慢解析。高频外部访问可调小或用 FQDN(末尾加 `.`)。
- CoreDNS 副本不够 → 大集群 DNS 超时。NodeLocal DNSCache 可显著降延迟。

## 3. 服务连通性 —— headless + 客户端缓存的真实 bug
**现象**:story-api 调不通 story service,但**重新部署 story-api 就好了**。

**根因**:
- Headless Service(`clusterIP: None`)DNS 直接返回后端 Pod IP 列表。
- story-api 启动时解析并**缓存**了 story service 的 Pod IP;story service 后来重新部署,Pod IP 变了,但 story-api 还用旧 IP/旧连接 → 调不通。
- "重新部署 story-api 就好" = 重启让它重新解析拿新 IP。

**解法**:
- 普通服务用 **ClusterIP**(稳定 VIP),别无脑用 headless。
- 客户端(尤其 gRPC)配 **DNS 重新解析 / 重连**(gRPC 用 `dns://` resolver + keepalive)。
- 或用 Service Mesh 处理服务发现。

## 4. 端口占用排查 —— 进容器看 Node 网络
```bash
# 起一个共享宿主机网络栈的 debug Pod
kubectl run netdebug --rm -it \
  --image=nicolaka/netshoot \
  --overrides='{"spec":{"hostNetwork":true,"nodeName":"目标节点名"}}' \
  -- bash

# 进去后看到的就是 Node 的网络
ss -tnlp        # 看端口占用(比 netstat 新)
lsof -i :8080   # 谁占了 8080
ip addr / ip route
```
`hostNetwork: true` 让 Pod 直接用宿主机网络命名空间 → 能看到节点真实端口和拓扑。
